| **Module Name** | `attendance-import` |

## Supplementary Analysis

### Levenshtein Extraction Strategy

The `levenshtein()` function currently lives in `clients/gemini/src/before_model.ts` (lines 25-50). This is a pure algorithmic function with no dependencies. The plan is to **copy** (not move) it into `packages/core/src/matching/levenshtein.ts` and export it from core's barrel. The Gemini sidecar continues to use its local copy for now -- refactoring the sidecar to import from core is a separate concern because `clients/gemini/` has its own `package.json` and build pipeline, and adding a workspace dependency there has broader implications.

### Name Matching Pipeline Design

The matcher in `packages/core/src/attendance/name-matcher.ts` should implement this pipeline:

1. **Persistent map lookup** -- lowercase the Zoom display name, look up in `zoom-name-map.json` entries. If found, return the Canvas user ID directly (skip fuzzy matching).
2. **Exact case-insensitive match** -- compare Zoom name against Canvas roster `user.name` and `user.sortable_name`.
3. **Levenshtein fuzzy match** -- compute normalized Levenshtein distance (`edit_distance / max(len_a, len_b)`). Auto-accept if distance is below a configurable threshold (suggest 0.25 for high confidence). Flag as ambiguous if between 0.25 and 0.5.
4. **Unmatched** -- any name that doesn't match in steps 1-3.

The matcher takes a roster (`Array<{ user_id: number, name: string, sortable_name: string }>`) and a list of Zoom participants (`Array<{ name: string, duration: number }>`), plus the persistent map contents. It returns a structured result with `matched`, `ambiguous`, and `unmatched` arrays.

### In-Memory Session State

The `parse` action stores its result in a module-scoped variable inside `packages/teacher/src/tools/attendance.ts` (same pattern as how `secureStore` is module-scoped in `index.ts`). The `submit` action reads from this variable and clears it after successful submission. This avoids temp files and keeps the parse-submit lifecycle within a single MCP session.

### Grade Submission API

Canvas grade submission uses: `PUT /api/v1/courses/:course_id/assignments/:assignment_id/submissions/:user_id` with body `{ submission: { posted_grade: "10" } }`. The `posted_grade` field accepts a string representation of the score. The new `gradeSubmission` function in `packages/core/src/canvas/submissions.ts` wraps this call.

### Risk: CSV Format Variations

Zoom CSV format can vary by Zoom version and locale. The parser should be defensive:
- Handle BOM at file start (common in Windows-generated CSVs)
- Handle both `\r\n` and `\n` line endings
- The `Name (original name)` column header is the canonical identifier; fall back to positional parsing if header names vary slightly

### File I/O Paths

Both `zoom-name-map.json` and `attendance-review.json` live alongside `config.json` in `~/.config/mcp/canvas-mcp/`. The `ConfigManager` already knows this base directory. The attendance module should derive the path from the same base rather than hardcoding `~/.config/mcp/canvas-mcp/`, to respect the `--config` flag's directory.
